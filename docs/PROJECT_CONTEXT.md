# PROJECT_CONTEXT

## 1. SYSTEM_OVERVIEW
- Product: multi-user financial control web app with authenticated session, encrypted state, and user-scoped backups.
- Core scope:
  - Authentication, profile, and per-user encrypted persistence.
  - Month operation (`Mês Atual`) with unified outflow model.
  - Dashboard analytics.
  - Patrimony (accounts, movements, transfers, variation).
  - Historical month comparison.
  - Developer admin area for technical audit/backup restore.
- Financial philosophy:
  - Competence month (`mês do lançamento`) is separated from cash flow (`quando o dinheiro saiu`).
  - Credit-card launches and card bill are separate entities.
  - Past months are immutable by default for recurrence propagation.
- Main entities:
  - `user`, `session`, `appState`, `month`, `outflow`, `outflowCard`, `cardBill`, `income`, `projectIncome`, `financialGoal`, `patrimonioAccount`, `patrimonioMovement`, `tag`, `category`, `backup`.

## 3. USER_MODEL
- User model fields include:
  - `id`, `username`, `email`, `passwordHash`, `encryptionSalt`, permissions, backup/login metadata.
- Session user payload includes:
  - `id`, `username`, `displayName`, `fullName`, `permissions`.
- State isolation model:
  - Every read/write is scoped by session user id.
  - Encrypted payload includes ownership validation.
  - Backup read/restore validates ownership.
- New users:
  - Created through `/api/auth/register`.
  - Initial state generated by `buildFreshUserAppState()`.
  - New architecture is default for all new users.
- Default initial behavior:
  - Unified outflow model enabled.
  - Default categories with emoji are preloaded.

### REGRAS CRITICAS
- Never share user data across users.
- Never reuse global mutable state across users.

## 4. AUTH_AND_SECURITY
- Login:
  - Email + password.
  - Session cookie `fin.sid`.
  - Optional remember-me cookie `fin.remember`.
- Password hash:
  - PBKDF2 SHA-512.
- Session security:
  - `httpOnly`, `sameSite=lax`, `secure` in production.
- Route protection:
  - `requireAuth`, `requireDeveloper`, `requireCsrf`.
- Request hardening:
  - Security headers, CSRF token, rate limit.
- Data encryption:
  - User state encrypted at rest with AES-256-GCM.
  - Key derived from user credentials/session context.
- Limitation:
  - Not zero-knowledge while user is authenticated.

## 5. DATA_PERSISTENCE
- Primary store:
  - JSON encrypted app-state per user (default backend).
  - transactional SQLite state store (`server/sqlite-app-state-store.js`) available behind `FIN_STATE_BACKEND=sqlite`.
  - backend selection centralized in `server/state-store-factory.js`.
  - compatibility mirror: when SQLite backend is active, writes are mirrored to JSON store to keep backup/audit tooling behavior compatible.
- Read/write APIs:
  - `readUserAppState(userId, encryptionKey)`
  - `writeUserAppState(userId, state, encryptionKey)`
- Integrity controls:
  - ownership check in payload.
  - migration-compatible normalization.
  - server-side defensive state sanitization before persistence:
    - non-object root state is coerced to safe object.
    - `finData` is guaranteed array.
    - `finStateSchemaVersion` is guaranteed (`'3'` default).
  - payload-size guard on state write (`MAX_STATE_BYTES = 15MB`) to prevent oversized/corrupted writes.
  - explicit schema version in app-state:
    - storage key: `finStateSchemaVersion`.
    - persisted metadata in `state.json` payload: `schemaVersion`.
    - backend schema constant source:
      - `server/state-schema.js`
      - `DEFAULT_FIN_STATE_SCHEMA_VERSION = '3'`
  - startup archive of orphaned legacy artifacts.

## 6. BACKUP_SYSTEM
- Modes:
  - Manual backup (user-triggered).
  - Automatic backup (login/activity policy).
- Storage layout:
  - `data/user-backups/<UserLabel__userId>/index.json`
  - `data/user-backups/<UserLabel__userId>/items/<backupId>/state.json`
- Retention:
  - user backup retention is non-pruning by default (no automatic loss by inactivity or backup count).
  - optional cap can be enabled with env `FIN_MAX_BACKUPS_PER_USER` (`>0`).
  - when cap is enabled, oldest backups are pruned by created date after each new backup.
- Restore:
  - only same-user backup is restorable.
  - pre-restore safeguards are enforced.
  - backup id validation hardened (`^[a-zA-Z0-9_-]{6,120}$`) before path resolution.
  - restore/revalidate path is constrained to the authenticated target user backup directory.
- Validation:
  - backup file schema + ownership + checksum validation.
  - backup id format validation before file-path resolution.
  - restore/revalidate path is constrained to user backup root.

### Deleted-user archives retention
- `data/deleted-users` purge is disabled by default.
- Optional timed purge can be enabled with env `FIN_DELETED_USER_RETENTION_DAYS` (`>0`).

### REGRAS
- Never restore backup into another user.
- Never overwrite user state without explicit restore action.

## 7. CORE_FINANCIAL_MODEL
- Canonical separation:
  - launch (`gasto/compromisso`) != payment settlement.
- Top month metrics (unified mode):
  - `Renda total`.
  - `Despesas planejadas para o mês`.
  - `Despesas do mês`.
  - `Resultado` (`Renda total - Despesas do mês`).
- `Já saiu`:
  - no longer a top card.
  - rendered in `Resumo` footer context inside `Compromissos do mês`.
- Inclusion controls:
  - checkboxes decide if item participates in primary totals.
  - defaults are marked as included.

## 8. CREDIT_CARD_MODEL
- Card is independent entity:
  - `month.outflowCards[]`.
- Bill is independent entity:
  - `month.cardBills[]`.
- Outflow launch linked to card:
  - `outflow.outputKind === 'card'` and `outputRef === card.id`.
- Anti-duplication rule:
  - card launches do not count as primary totals when bill is authoritative.
  - never sum bill + same launches in same primary metric path.
- Past creation exception (explicit rule):
  - creating a card in a past month must propagate that card from that month forward.
- Delete rule:
  - deleting card affects current selected month forward only.
  - past months remain unchanged.
- Delete flow with linked data:
  - modal offers transfer-to-other-card or `Excluir dados`.
  - `Excluir dados` requires explicit second confirmation.
- Future bill forecast:
  - if month has no manual bill value, recurring card launches can compose initial forecast.
  - first manual bill input overrides forecast for that month.

### REGRAS CRITICAS
- Never duplicate card bill and card launches in main sums.
- Never mutate historical card state in months prior to selected month during delete/update flows.

## 10. RECURRENCE_ENGINE
- Supported recurrence types:
  - fixed outflows.
  - recurring spends.
  - fixed incomes (`renda` with `recurringFixed=true`).
  - installments.
- Identity:
  - `recurringGroupId`.
  - `installmentsGroupId`.
- Propagation policy:
  - only current/future may propagate.
  - past never updated by forward propagation.
  - propagation applies only changed fields for recurring edit scope `forward`.
- Runtime synchronization:
  - `ensureUnifiedRecurringFutureCoverage()` keeps future months consistent.
  - existing future months are rechecked/synchronized.
- Editing scope UX:
  - custom app modal (not browser confirm) with:
    - this month only
    - this month forward
    - cancel
- Fixed income propagation rules:
  - fixed income receive field is day-based (`DD`) for recurring fixed income.
  - default receive month for recurring fixed income is the next month (`competence month + 1`).
  - manual full-date override (`DD/MM/YY`) is accepted and respected as explicit receive month/year.
  - implementation source of truth for receive-day parsing/render/sort:
    - `IncomeDateRules.getRecurringIncomeReceiveDay`
    - `IncomeDateRules.getIncomeReceiveDateLabel`
    - `IncomeDateRules.getIncomeReceiveDateSortValue`
  - propagation eligibility helper extracted:
    - `MesAtualRecurrence.canPropagateRecurringFromMonth`
  - creating fixed income in month creates/updates same recurring group in future months.
  - edit/delete in past month does not propagate.
  - edit/delete in current/future month asks scope and can propagate forward only.

  - eligible month list is deduplicated by month key/id to prevent duplicated-month inflation in cards/charts.
- Pie chart:
  - only valid real categories.
  - never use type labels as category.
  - categories <1% grouped into `Categorias menores`.
  - card type labels must not appear as category artifacts.
- Line chart (`Gastos vs Renda`):
  - must start at zero baseline (`min: 0`, `beginAtZero` equivalent behavior).
- Metrics consistency:

- Month top-card variation semantics:
  - for expense cards (`Despesas planejadas`, `Despesas do mês`) variation color is inverted:

    - positive variation = red (worse)

## 20. BILL_IMPORT_FLOW
- Entry point:
  - `Compromissos do mÃªs` action: `Adicionar por fatura`.
  - Opens `modalBillImport`.
  - Accepts two ingestion modes:
    - JSON file upload
    - pasted AI response text (extracts JSON from fenced block or raw text)
- AI role:
  - AI acts only as pre-processor.
  - App state remains source of truth.
  - Import is always reviewed before write.
- Context export:
  - file: `finance_import_context.json`
  - format: `finance_import_context_v1`
  - includes cards, categories, tags reference, strict system rules and expected output contract.
- Prompt export:
  - generated by `BillImportUtils.buildExternalAiPrompt`.
  - enforces:
    - response must be a single ` ```json ... ``` ` block
    - no text outside JSON block
    - no new categories
    - no tag creation/suggestion
    - `tag` always null
    - no bill-total row as transaction
    - ignore reimbursements/estornos from card bills
    - uncertain category -> `category: null` + `needs_review: true`
- Accepted AI output:
  - `format = finance_import_v1`
  - `version = 1`
  - payload supports:
    - `items[]`
    - `transactions[]`
    - `cards[].months[].transactions[]`
  - item core fields:
    - `date`, `description`, `amount`, `card_id|card_name`, `month_id?`, `category`, `suggested_categories?`, `confidence?`, `needs_review?`, `tag`, `warnings?`, `source_excerpt?`
- Validation:
  - schema blocking errors:
    - invalid JSON/format/version/required fields
  - semantic checks:
    - invalid date/amount/card/category
    - date is normalized before month/card/category semantic checks
    - month inference by transaction date (`DD/MM/YY`) when `month_id` is missing
    - parser accepts year-first variants (`YY/MM/DD`, `YYYY/MM/DD`) and normalizes to `DD/MM/YY`
    - inferred/provided month id is reconciled against existing user months by date month/year to avoid false "month does not exist" warnings caused by legacy id formats
    - card inference fallback from payload headers/hints (example: `cards_identified: ["XP"]`)
    - likely bill total rows
    - reimbursement/estorno rows are ignored automatically
    - low-confidence signals
  - system checks:
    - probable duplicates via card+date+amount+description similarity against existing launches
    - missing month warning (month can be created at import)
  - per-item status:
    - `valid`, `warning`, `needs_review`, `error`
- Category and tag rules:
  - category must match existing user categories (or remain empty for manual review).
  - AI cannot create categories.
  - AI cannot set tags; incoming tags are discarded.
  - user can set tag manually during review.
- Review stage (`modalBillImportReview`):
  - mandatory before import.
  - grouped by month + card.
  - editable fields:
    - include, date, description, amount, card, category, tag.
  - tags are shown as selectable options from existing user tags.
  - edits update canonical in-memory `reviewState.items[]` directly (no separate shadow copy).
  - status recalculated on the same canonical item after each edit.
  - date edit re-normalizes date and re-resolves `monthId` on the canonical item.
  - card edit stores final `cardId` and resolved `cardName` on the canonical item.
  - import enabled only when selected items have no `error` and no `needs_review`.
- Final import write:
  - import source is canonical edited `reviewState.items[]` (not raw payload).
  - inserts outflows as card launches:
    - `type=spend`, `outputKind=card`, `countsInPrimaryTotals=false`
  - does not import bill totals as launches.
  - month resolution order in import:
    - normalized date (`dd/mm/aa`) -> month/year
    - normalized `monthId` candidate
    - current selected month fallback
  - import ensures target month exists (`ensureMonthExists`) even when `monthId` string is non-canonical.
  - ensures selected card exists in target month (cloned from known template when absent).
  - syncs month through existing unified/legacy compatibility pipeline.
  - logs import metadata in `month.billImportHistory[]`.
- Module architecture:
  - `public/app/modules/import-bills-schema.js`
  - `public/app/modules/import-bills-utils.js`
  - `public/app/modules/import-bills-review.js`
  - `public/app/modules/import-bills.js`

## 21. FINANCE_CALENDAR_MODULE
- Entry:
  - month header icon opens `modalFinanceCalendar`.
  - modal is floating, draggable, non-blocking background, and does not close on outside click.
- Rendering:
  - grid render is delegated to `calendar-render.js`.
  - event persistence helpers are in `calendar-events.js`.
  - date/intensity/math helpers are in `calendar-utils.js`.
  - orchestration and modal actions are in `calendar.js`.
- Month entity extension:
  - `month.calendarEvents[]`:
    - `{ id, name, startDate, endDate, budget, color, tagId }`
  - `tagId` is tag string reference; categories are never used as event key.
- Day coloring rule:
  - based only on variable outflows (`type=spend`, non-recurring, non-fixed).
  - fixed/recurring/predictable commitments do not affect day heatmap.
  - fixed outflows with explicit month/year are rendered only in their real billing month (no day-only fallback across month boundaries).
